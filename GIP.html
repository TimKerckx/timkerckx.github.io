Geïntegreerd project

SDLC Binnen onze gip:

SDLC van ons project: Hier gaan we over de System Development Life Cycle voor ons geïntegreerd project. We hebben elk van de 6 fases uitgelegd wat en hoe we ze gaan uitwerken.

Planning-analyse: In de aula werden er groepen gekozen, we zijn dan samengekomen tot: Thomas, Tim, Jaimie, Joni en Karel. Nadien hebben we als groep besloten welke communicatiemiddelen we gaan gebruiken voor aan ons project te werken (discord, messenger, ...). We hebben besproken op welke momenten we hieraan zouden werken. Nadien is beslist om samen te komen op de campus zodat we ons zo goed mogelijk konden focussen op het project. Bij onze eerste samenkomst hebben we dan ook geanalyseerd hoe we gaan werken, en gebrainstormd over de toekomstige taken die ons te wachten staan binnen het project. Er is ook een taakverdeling opgesteld om het werk evenredig te verdelen. Dit ook zodat alle leden binnen het project te allen tijde weten waar ze aan kunnen werken.

Design: Na de planningsanalyse zijn we op de eerste samenkomst aan het ontwerp begonnen. We hebben samen nagedacht over wat we allemaal gaan toepassen, en ervoor gezorgd dat iedereen dit begreep. We besloten om de database te hosten, omdat dit voor ons de meest praktische oplossing bleek te zijn. We vonden dit een belangrijke stap vooraleer we aan het project begonnen zijn. Dit zodat we wisten dat iedereen op hetzelfde niveau zit, en dat we allemaal evenveel bijdragen leveren aan het project.

Develop: Na de designfase hebben we hetgeen toegepast dat in de projectlessen is aangehaald. Dit hebben we ook in groep gedaan om zo een goede samenwerking op punt te stellen. Vooraleer we begonnen met het schrijven van de code, hebben we testen geschreven voor ons programma. Dit omdat het ons achteraf veel werk zou besparen. Dit hebben we eveneens in groep gedaan, om een snelle voorruitgang te boeken. Zo wist iedereen op wat er getest ging worden, wat het nut is van deze testen en ook waar we in te toekomst rekening mee moeten houden. Nadien hebben we ervoor gezorgd dat iedereen hun eigen taak heeft bij het project. Dit werd telkens nagekeken door een ander teamlid.

Implementation: Nadat de meeste code is geschreven in de develop fase, gaan we het project in productie zetten. Ons project zal dan ook gehost worden op een onlineserver. Hiervoor moeten we zorgen dat er voldoende security maatregelen en controles zijn. Zodat zowel de server van de site en de database veilig online staan.

Maintain: Aan maintain denken we momenteel dat we niet veel gaan doen. We verwachten dat dit project alleen maar dit semester is en we dus echter niets voor maintain moeten of kunnen doen. Echter, als dit niet zo is kunnen we altijd een persoon aanduiden voor de applicatie te onderhouden.


OWASP binnen onze gip:

Hoe kan OWASP helpen met je geïntegreerd project?
Ons doel is om elk punt van de OWASP top tien te overlopen en voor elk van deze punten de nodige veiligheidsmaatregelen te noteren, en toe te passen in ons project.

Punt 1: Injection. Hier gaan we zorgen voor server-zijde input validatie: dit wil zeggen dat we alle inputvelden van onze applicatie gaan ter beschikking stellen voor de gebruikers. Deze input gaan we controleren op foutieve karakters en of SQL commando strings,… Zodat aanvallers geen onbedoelde commando’s kunnen sturen.

Punt 2: Broken authentication We gaan onze applicatie van een 2 factor authentication systeem voorzien. Dit zorgt ervoor dat bij spraken van een gegevenslek de aanvaller niet kan inloggen met een gebruikersnaam en wachtwoord combinatie. Dit omdat hij nog een tweede code moet ingeven die bijvoorbeeld op de gsm/tablet van de gebruiker gegenereerd zal worden. Ook gaan we een limiet zetten op het aantal keer dat men kan proberen inloggen met een foutieve gebruikersnaam en wachtwoord.

Punt 3: Sensitive data exposure We hebben de data van onze applicatie beschermd door hem te hosten achter een firewall. Deze data is enkel beschikbaar op 1 willekeurig poortnummer en alleen maar beschikbaar met de juiste login gegevens.

Punt 4: XML External Entities (XXE) Voor ons project zullen we JSON data format gebruiken.

Punt 5: Broken Access Control Bij elke toegangscontrole zal er gekeken worden wie de gebruiker is, ook kijken we telkens of hij de juiste bevoegdheid heeft. Als hij geen toegang heeft krijgt deze een aangepaste en correcte foutmelding. We gaan eenmaal een toegangscontrolemechanisme implementeren en hergebruiken doorheen de hele applicatie.

Punt 6: Security Misconfiguration Als bij het uitvoeren van een taak in de applicatie een bepaalde error verschijnt omdat de gebruiker in kwestie een actie onderneemt waar hij niet voor bevoegd is, of deze een fout geeft in het systeem. Zullen we zorgen voor de juiste veralgemeende foutmelding. Dit in front end en back end niveau.

Punt 7: Cross-Site Scripting Hier beschermen we ons op dezelfde manier als bij injection.

Punt 8: Insecure Deserialization Beschermen hiertegen kan door andere data-formaten te gebruiken dan de originele standaard. Dit doen we door JSON te gebruiken. Ook Gebruiken we een firewall die schadelijke of onbevoegde onveilige deserialisatie kan detecteren.

Punt 9: Using Components With Known Vulnerabilities Hier zorgen we ervoor dat we alle niet-gebruikte dependencies, onnodige toepassingen, componenten en documenten verwijderen uit de applicatie. Dit zodat deze niet tegen onszelf gebruikt kunnen worden.

Punt 10: Insufficient Logging and Monitoring We hebben ons voorgenomen om logging/monitoring te doen op onze applicatie zodat aanvallen vroegtijdig ontdekt of onderschept kunnen worden. Ook hebben we een plan opgesteld voor als er toch een aanval gebeurt, zo kunnen we snel en op tijd handelen om de schade van de applicatie te beperken.
